早期软件应用程序

在早期，软件应用程序是作为一个单体设计和部署的。这时候的应用程序，单个实例执行了所有业务功能。

这些应用程序大多数是企业应用软件，主要用于企业内的简单任务，例如：会计和计费。它们的运维是可以接受短时间的停机，因为在那时，这些程序 24 * 7 可用性不是必需的。

新千年的颠覆

上世纪 90 年代未和 21 世纪初，技术上的突破打破了软件应用程序的限制。软件应用程序开始执行越来越复杂的任务，如商业智能、客户关系管理等。

为了有效地处理这种复杂性，这些应用程序的代码库在不断增长。除此之外，这段时间互联网的使用量在指数级增长，催生了一个全新的维度 - Web 应用程序。

这些 web 应用程序可以从任何角落，24 小时不间断地访问。这就要求它们的工作负载具有足够的高可用性。

云计算的崛起

随着应用程序的规模以及对它们全天候可用性需求的增长，如何在本地维护这些应用程序成为一个挑战。相反，按需计算的云平台成为了新的选择。

云计算平台提供了运行应用程序即服务所需的物理硬件资源。这些组织承若其基础设施具有高可用性和可靠性，并根据使用多少资源以及多长时间来向客户收费。目前流行的云服务提供商有 AWS、Microsoft Azure 和 Google Cloud。

开发和部署问题仍未解决

即使云计算解决了物理硬件上的问题，但应用程序生命周期中仍有许多未能解决的问题，例如开发和部署。

  • 大量的构建时间。大型的代码库导致了耗费大量的构建时间，即会产生大量的停机部署时间。停机是不可接受的，所以高可用性成为了关键要求。

  • 交付缓慢。大型代码库的另一个问题是对开发速度的影响。由于所有的代码都包含在一个实例中，因此团队中一位开发人员的无效代码都可能影响其他开发人员的工作。在 2000 年代初期，随着敏捷软件开发方法的出现，使应用可以持续改进和快速交付。

  • 对故障的应变能力差。单体式 Web 应用程序对故障的处理非常差。所有业务功能都由单个实例处理。执行单个业务功能发生崩溃，可能会导致整个应用程序的崩溃。

微服务

为了应对上述架构上的问题，出现了一种新的架构 - 微服务。

在这种架构中，应用程序被设计为一组解耦合的服务，而不是单个的整体应用程序。每一个服务都会负责一个独立的业务功能。这些服务在必要时相互通信，并共同满足所有的业务需求。

将应用程序分解为不同的小型服务可以更轻松地理解、开发和测试。解决了上述单体式架构的问题。

  • 易于部署。相对于传统单体式应用，基于微服务的应用更加模块化且小巧，利于持续的交付和部署。

  • 高度可扩展。随着某些服务的不断扩展，可以跨多个服务器和基础架构进行部署。

  • 出色的弹性。只要确保正确构建，这些独立的服务就不会彼此影响。这意味着，一个服务出现故障不会导致整个应用的下线，这一点与单体式应用不同。
